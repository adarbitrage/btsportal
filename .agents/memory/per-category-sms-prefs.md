---
name: Per-category SMS preferences
description: How finer-grained (non-master) SMS opt-outs are enforced
---
Members have a master `users.sms_opt_in` plus finer-grained per-category flags: `ticket_reply_sms_opt_in` (default true), `security_sms_opt_in` (true), `billing_sms_opt_in` (true), `coaching_sms_opt_in` (true), `content_sms_opt_in` (false).

**Why:** members wanted to keep security/billing texts but silence the text on every support reply (etc.), without opting out of SMS entirely.

**How to apply:** the category gate ALWAYS lives in the CALLER that queues the SMS — `CommunicationService.queueSms` / `sendSmsDirect` is channel-generic and only re-checks the master `smsOptIn`, NEVER category flags. Each caller must `select` the master + its category flag and check `smsOptIn && <cat>SmsOptIn && phone` before queueing. Email always sends regardless. Live enforcement points: ticket-reply → `sendTicketReplyNotification` in `tickets.ts`; billing (purchase_confirmation + payment_failed) → `webhook-handler.ts`; security (flexy_password_reset) → `admin-apps.ts` (adds a `category_opted_out` skip reason alongside `not_opted_in`/`no_phone_on_file`); coaching (coaching_reminder, 1h call window) + content (new_content_alert, scans `announcements` created in last 24h) → `scheduled-comms.ts` (per-member dedup via `checkAndRecordSend`; coaching recipients resolved by entitlement join users↔user_products↔products with `entitlement_keys::text LIKE %requiredEntitlement%`). Content recipients are simply opted-in (contentSmsOptIn default false) — not entitlement-gated.

**SMS template seeding trap:** new SMS templates added to the `smsTemplates` array in `seed-templates.ts` ONLY go live via `seedCommunicationTemplates()` (TRUNCATE email_templates+sms_templates then re-insert). Unlike email there is NO `ensureRequired*` for SMS on boot, so adding an SMS template is a no-op in a running prod (queueSms returns `template_not_found` → skipped) until a full re-seed runs.

To add/wire a category: schema bool col in `lib/db/src/schema/users.ts` (+ dev ALTER, no companion .sql needed — additive col w/ default) + expose on MemberProfile + PatchMemberProfileBody/Response in `lib/api-spec/openapi.yaml` then regen codegen + persist in `onboarding.ts` PATCH /members/me/profile + return on members.ts GET /members/me + gate the relevant caller + add a sub-toggle in portal `Account.tsx` (state/reset-effect/notifDirty/save-payload + a Switch disabled when `!smsOptIn`).

**Codegen gotcha:** `pnpm --filter @workspace/api-spec run codegen` (orval over the ~10k-line openapi.yaml) takes >2min and exceeds the 120s tool cap; if it can't complete, hand-edit the deterministic generated files to match the existing field's pattern: `lib/api-zod/src/generated/{api.ts, types/*.ts}` (zod schemas + TS types) and `lib/api-client-react/src/generated/api.schemas.ts` (TS interfaces — MemberProfile, PatchMemberProfileBody, PatchMemberProfileResponse). Then rebuild libs sequentially: `tsc -b lib/db; tsc -b lib/api-zod; tsc -b lib/api-client-react` (one at a time — the combined build also times out). Full-package tsc (`typecheck` per package) also exceeds 120s; run the `typecheck` workflow with a high workflow_timeout and read results via refresh_all_logs.
