AMENDMENT to the tiered-onboarding spec (supersedes TA1; amends TB1; TA2 and TB2 unchanged). Grounded in the completed diagnostic.

TA1 (revised) — Phase 0 first: variant-aware step contract refactor.
Phase 0: Collapse the three step-list definitions (OnboardingLayout STEPS/ROUTES, routes/onboarding STEP_NAMES, onboarding-advancement ONBOARDING_STEP numbers) into ONE shared step-contract module: an ordered array of step KEYS per variant (FULL: welcome, profile, book_kickoff, book_partner_call, seven_pillars, first_call; LAUNCHPAD: welcome, profile, book_kickoff, seven_pillars). All advancement becomes key-based: advanceIfOnStep(memberVariantArray, fromKey) resolves positions at call time; the kickoff-booked advance is "advance past book_kickoff", partner-booked "advance past book_partner_call", completion "member is at the LAST key of THEIR variant" — no absolute numbers anywhere. Stepper, guards, prerequisites, client-advanceable set ({welcome, profile, seven_pillars}) all read the shared module. Numbers persist in the DB as positions within the member's variant array (no schema change), but code never hardcodes them.
Phase 1: The tier resolver (resolveOnboardingVariant via PRODUCT_RANK — the slug-keyed server map is the SINGLE authority; do not use the entitlement-keyed label map or the portal duplicate for tier decisions) + creation-time defaults: webhook/creation path sets variant 'none' members onboardingComplete=true at creation (and fires the front-end sequence handoff there); launchpad/full start at their variant's first step. The portal learns the member's variant from the auth/me payload (derived server-side from entitlements — no new stored column).
LaunchPad completion trigger: completing seven_pillars (the last key of that variant) completes onboarding.
