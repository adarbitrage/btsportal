# BTS Portal — New Member Onboarding Flow Build Spec

**Priority:** After Auth + ThriveCart Webhook
**Status:** Not started
**Depends on:** Custom Auth System, ThriveCart Webhook Integration
**Reference:** BTS PRD v2.0, Sections 4.3, 6.3, 16

---

## Context

The `users` table already has an `onboarding_complete` field (boolean, defaults to `false`). When a new member logs in for the first time — whether they registered manually or were auto-provisioned via ThriveCart webhook — they need to be routed through a structured onboarding experience before they can access the main dashboard.

The onboarding flow has two jobs: (1) collect required information and legal agreements, and (2) give the new member a clear, confident path to their first win inside the portal. A member who completes onboarding should know exactly what they bought, what's available to them, and what to do first.

---

## Routing Logic

### Global Route Guard

Every authenticated page in the portal must check `onboarding_complete` before rendering. This is a single check at the top of the protected route wrapper.

```
IF user is authenticated AND onboarding_complete === false:
  → Redirect to /onboarding (regardless of what URL they requested)
  → The ONLY pages accessible are /onboarding/* and /api/auth/logout

IF user is authenticated AND onboarding_complete === true:
  → Allow normal routing (dashboard, training, coaching, etc.)
  → Block access to /onboarding (redirect to /dashboard if they try to go back)
```

**Important:** The onboarding state is fetched from `/api/members/me` on initial auth check. Don't rely on a local flag that could get out of sync — always trust the server.

---

## Onboarding Flow (5 Steps)

The onboarding is a multi-step wizard. Each step is a screen within `/onboarding`. Progress is saved after each step so the member can resume if they close the browser mid-flow.

The URL structure:
```
/onboarding/welcome          → Step 1
/onboarding/documents        → Step 2
/onboarding/profile          → Step 3
/onboarding/orientation      → Step 4
/onboarding/quick-start      → Step 5
```

A progress indicator (step dots or progress bar) is visible at the top of every step showing 1 of 5, 2 of 5, etc.

Members cannot skip steps or jump ahead. They must complete each step sequentially. The back button is available on steps 2–5 to review previous steps (read-only if already submitted).


**Scroll-to-read enforcement:**
- Each document is in a scrollable container with a fixed height (~300px)
- The checkbox for each document is disabled until the user scrolls to the bottom
- Use an `IntersectionObserver` or scroll position check on a sentinel element at the bottom of each document
- This is a UX pattern for legal compliance — it doesn't guarantee they read it, but it demonstrates they had the opportunity

**Document content:**
- Stored in the database or as markdown files, editable via admin panel
- Rendered as formatted HTML inside the scrollable container
- Version-tracked: if the document is updated after a member signs, their signature still references the version they agreed to

**Signature:**
- For v1: typed signature (member types their full legal name into a text input)
- Future enhancement: canvas-drawn signature
- The typed name must match the name on their account (case-insensitive) OR be any non-empty string (depending on legal requirements — for now, just require non-empty)

**On submit:**

```
POST /api/documents/sign
{
  "documents": [
    {
      "type": "membership_agreement",
      "version": "1.0",
      "agreed": true
    },
    {
      "type": "terms_of_service", 
      "version": "1.0",
      "agreed": true
    }
  ],
  "signature": "John Michael Doe",
  "signed_at": "2026-03-12T14:30:00Z"
}
```

**Backend logic:**
1. Validate all documents are agreed to and signature is non-empty
2. For each document:
   - Generate a PDF containing: document content, member name, email, signature, date/time, IP address
   - Store PDF in Cloudflare R2 (path: `documents/{user_id}/{document_type}_{timestamp}.pdf`)
   - Insert record into `signed_documents` table
3. TODO: Email a copy of signed documents to member via SendGrid (log for now)
4. Save onboarding step progress

**If documents are already signed** (member went back to this step):
- Show a "You've already signed these documents" message with the date signed
- Show a "View your signed documents" link
- "Continue" button is immediately active

## API Endpoints

### New Endpoints

#### `PATCH /api/members/me/onboarding`

Saves progress on each onboarding step.

**Request body:**
```json
{
  "step": "welcome",       // "welcome" | "documents" | "profile" | "orientation" | "quick_start"
  "completed": true
}
```

**Logic:**
1. Map step name to step number (welcome=1, documents=2, profile=3, orientation=4, quick_start=5)
2. Verify the member hasn't already completed onboarding (`onboarding_complete` must be false)
3. Verify the requested step is the next step in sequence (can't skip ahead)
4. Update `onboarding_step` on user record
5. If step is "quick_start": also set `onboarding_complete = true` and `onboarding_completed_at = NOW()`
6. Return 200 with updated user profile

#### `GET /api/members/me/onboarding`

Returns the member's current onboarding status and step.

**Response:**
```json
{
  "onboardingComplete": false,
  "currentStep": 3,
  "stepsCompleted": {
    "welcome": true,
    "documents": true,
    "profile": false,
    "orientation": false,
    "quick_start": false
  },
  "documentsSignedAt": "2026-03-12T14:30:00Z"    // null if not signed
}
```

#### `GET /api/onboarding/orientation`

Returns the orientation items filtered by the member's entitlements.

**Response:**
```json
{
  "accessible": [
    {
      "title": "Foundational Training",
      "description": "6 modules of video + text training...",
      "icon": "book-open",
      "navLink": "/training"
    }
  ],
  "locked": [
    {
      "title": "Group Coaching Calls",
      "description": "Weekly Q&A and strategy sessions...",
      "icon": "video",
      "unlockedBy": "BTS 3-Month Mentorship"
    }
  ]
}
```

## Frontend Components

### OnboardingLayout

A shared layout wrapper for all 5 onboarding steps:
- Clean, focused design (no sidebar navigation — just the BTS logo and a progress indicator)
- Progress bar or step dots at the top
- Consistent padding, max-width container (700–800px centered)
- Mobile-responsive
- No access to the main portal navigation (keep them focused)

### ProgressIndicator

Shows steps 1–5 with:
- Completed steps: filled circle with checkmark
- Current step: filled circle, highlighted
- Future steps: empty/gray circle
- Step labels below each dot (optional on mobile)

### ScrollableDocument

Reusable component for the document signing step:
- Fixed-height scrollable container
- Renders markdown/HTML content
- Tracks scroll position
- Fires `onScrolledToBottom` callback to enable the agreement checkbox
- Shows a subtle "Scroll to continue" indicator if they haven't reached the bottom

### ProductOrientationCard

Reusable card for the orientation step:
- Icon, title, description, link
- Two variants: "accessible" (green check, clickable link) and "locked" (lock icon, upgrade prompt)

## Conditional Logic Details

### Quick Start — Dynamic Content Rules

The "first mission" message on Step 5 depends on the member's highest product level:

```
IF member has any Mentorship product:
  - Show next upcoming coaching call date/time
  - Primary CTA: "Start Module 1 so you're prepared for your first call"
  - Secondary links: Training, Call Schedule, Community

IF member has LaunchPad (no Mentorship):
  - Primary CTA: "Start Module 1, then explore the Software & Tools"
  - Secondary links: Training, Tools, AI Assistant

IF member has Front-End only:
  - Primary CTA: "Start Module 1: The Affiliate Marketing Landscape"
  - Secondary links: Training, AI Assistant, Support
```

## Definition of Done

1. New members with `onboarding_complete: false` are always redirected to `/onboarding` regardless of requested URL
2. Members with `onboarding_complete: true` cannot access `/onboarding` (redirected to dashboard)
3. All 5 steps render correctly and enforce sequential completion
4. Document signing captures typed signature, timestamps, and IP address for both documents
5. Profile setup saves all fields including experience level, primary goal, timezone, and SMS opt-in
6. Orientation step dynamically shows accessible features vs locked upgrades based on the member's actual entitlements
7. Quick start step shows product-appropriate first action with correct coaching call data (if applicable)
8. Onboarding progress is saved per-step — closing the browser and returning resumes at the correct step
9. Completing Step 5 sets `onboarding_complete: true` and redirects to dashboard
10. Two demo users demonstrate both states: one who sees the full onboarding flow, one who goes straight to dashboard
11. Admin panel shows onboarding analytics and document management
