# BTS Portal — Public & Partner API Specification

**Priority:** Build alongside core features (API-first development)
**Status:** Not started
**Depends on:** Auth system, all feature modules
**Reference:** BTS PRD v2.0, all build specs

---

## Context

The BTS portal has internal API routes defined across 14 build specs. But the platform also needs a formal, documented, versioned API that third-party systems can consume — Zapier/Make automations, custom CRM integrations, analytics platforms, mobile apps, affiliate tools, white-label partners, and anything else that needs to read or write BTS data programmatically.

This spec defines the complete public API surface: authentication, versioning, rate limiting, every resource endpoint, outgoing webhooks (so third parties can subscribe to BTS events), error standards, and documentation infrastructure.

---

## API Design Principles

1. **RESTful** — resources as nouns, HTTP methods as verbs, predictable URL patterns
2. **JSON everywhere** — all requests and responses are `application/json` (except file uploads and webhooks)
3. **Versioned** — all endpoints prefixed with `/api/v1/`. Breaking changes get a new version.
4. **Consistent error format** — every error returns the same structure
5. **Pagination** — all list endpoints support cursor-based pagination
6. **Filtering** — list endpoints support query params for filtering and sorting
7. **Rate limited** — per-key limits with clear headers
8. **Idempotent** — PUT/DELETE operations are safe to retry
9. **Documented** — OpenAPI 3.1 spec auto-generates interactive documentation

## Authentication

### Three Auth Methods

The API supports three authentication methods for different use cases:

#### Method 1: API Keys (Server-to-Server)

For backend integrations (Zapier, Make, custom scripts, partner systems). API keys are tied to an admin user and inherit that user's role permissions.

```
Authorization: Bearer bts_live_sk_a1b2c3d4e5f6...
```

**Key format:** `bts_{environment}_{type}_{random}`
- Environment: `live` or `test`
- Type: `sk` (secret key, full access) or `pk` (publishable key, read-only)
- Random: 32-character alphanumeric string

**Key management:**
```sql
api_keys
  id              SERIAL PRIMARY KEY
  user_id         INTEGER REFERENCES users(id) NOT NULL  -- admin who owns this key
  name            TEXT NOT NULL              -- "Zapier Integration", "Analytics Dashboard", etc.
  key_hash        TEXT NOT NULL              -- bcrypt hash of the key (never store plaintext)
  key_prefix      TEXT NOT NULL              -- first 8 chars for identification (e.g., "bts_live")
  environment     TEXT NOT NULL              -- 'live' or 'test'
  key_type        TEXT NOT NULL              -- 'secret' or 'publishable'
  permissions     JSONB NOT NULL DEFAULT '["*"]'  -- granular permissions (see below)
  rate_limit_tier TEXT DEFAULT 'standard'    -- 'standard', 'elevated', 'unlimited'
  last_used_at    TIMESTAMP
  expires_at      TIMESTAMP                 -- optional expiry
  is_active       BOOLEAN DEFAULT true
  created_at      TIMESTAMP DEFAULT NOW()
  revoked_at      TIMESTAMP
```

**Key creation flow:**
1. Admin goes to Settings → API Keys → Create New Key
2. Selects name, environment, type, permissions, and optional expiry
3. System generates key, shows it ONCE (never shown again), stores hash
4. Admin copies key and configures it in the third-party system

#### Method 2: JWT Cookies (Browser/Portal)

For the member-facing portal frontend. Already implemented in the Auth build spec. Not used by third-party integrations.

```
Cookie: access_token=eyJhbGciOiJIUzI1NiIs...
```

#### Method 3: OAuth 2.0 (Partner Applications) — Future

For partner applications that act on behalf of members (e.g., a mobile app, a third-party dashboard, or a white-label integration).

Not built in v1. Spec reserved for future implementation. When added:
- Authorization Code flow with PKCE
- Scopes map to entitlement-like permissions
- Refresh token rotation
- Consent screen for member authorization

### Granular Permissions

API keys can be scoped to specific resources and actions:

```json
// Full access (admin)
["*"]

// Read-only access to members and training
["members:read", "training:read"]

// Webhook management only
["webhooks:read", "webhooks:write"]

// Commission data access
["commissions:read", "commissions:write", "affiliates:read"]
```

**Permission format:** `{resource}:{action}`

**Resources:** `members`, `products`, `training`, `coaching`, `community`, `support`, `chat`, `tools`, `commissions`, `communications`, `webhooks`, `admin`, `system`

**Actions:** `read`, `write`, `delete`, `admin`

The API middleware checks the key's permissions before allowing access to any endpoint. Missing permission → 403.

## Rate Limiting

### Tiers

| Tier | Requests/Minute | Requests/Hour | Requests/Day | Who Gets It |
|------|----------------|---------------|-------------|-------------|
| Standard | 60 | 1,000 | 10,000 | Default for all API keys |
| Elevated | 300 | 5,000 | 50,000 | Partners with approved integration |
| Unlimited | No limit | No limit | No limit | Internal services, super admin |

### Headers

Every API response includes rate limit headers:

```
X-RateLimit-Limit: 60
X-RateLimit-Remaining: 45
X-RateLimit-Reset: 1711065600
X-RateLimit-Tier: standard
```

When rate limited (429):

```json
{
  "error": {
    "code": "rate_limit_exceeded",
    "message": "Rate limit exceeded. Retry after 2026-03-22T10:00:00Z.",
    "retryAfter": "2026-03-22T10:00:00Z",
    "limit": 60,
    "window": "1m"
  }
}
```

### Implementation

Redis-backed sliding window counter keyed by API key ID. Each request increments the counter. Counter expires at the window boundary.

## Versioning

All API endpoints are prefixed with `/api/v1/`. When breaking changes are needed, a new version (`/api/v2/`) is created. The old version continues to work for a deprecation period (minimum 6 months).

**What constitutes a breaking change:**
- Removing a field from a response
- Changing a field's type
- Removing an endpoint
- Changing required parameters
- Changing error codes

**What does NOT constitute a breaking change (additive, safe):**
- Adding a new field to a response
- Adding a new optional parameter
- Adding a new endpoint
- Adding a new enum value

## Error Handling

Every error response follows the same format:

```json
{
  "error": {
    "code": "validation_error",
    "message": "Human-readable error description.",
    "details": [
      { "field": "email", "message": "Invalid email format" },
      { "field": "name", "message": "Name is required" }
    ],
    "requestId": "req_a1b2c3d4e5"
  }
}
```

### Standard Error Codes

| HTTP Status | Code | Meaning |
|-------------|------|---------|
| 400 | `bad_request` | Malformed request body or params |
| 400 | `validation_error` | Input validation failed (details array included) |
| 401 | `unauthorized` | Missing or invalid authentication |
| 401 | `token_expired` | JWT access token has expired |
| 403 | `forbidden` | Authenticated but insufficient permissions |
| 403 | `entitlement_required` | Member lacks required entitlement (includes `upgradeProduct`) |
| 404 | `not_found` | Resource does not exist |
| 409 | `conflict` | Resource already exists or state conflict |
| 409 | `duplicate` | Duplicate operation (idempotency check) |
| 422 | `unprocessable_entity` | Valid JSON but semantically invalid |
| 429 | `rate_limit_exceeded` | Too many requests |
| 500 | `internal_error` | Server error (details logged, not exposed) |
| 503 | `service_unavailable` | Temporary outage (retry with backoff) |

### Request IDs

Every request gets a unique `requestId` (UUID v4) returned in the response body and as a header (`X-Request-Id`). Include this in support requests for debugging.

## Pagination

All list endpoints use **cursor-based pagination** (not offset-based) for consistency and performance with large datasets.

### Request

```
GET /api/v1/members?limit=20&cursor=eyJpZCI6MTAwfQ==&sort=created_at:desc
```

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `limit` | integer | 20 | Results per page (max 100) |
| `cursor` | string | null | Opaque cursor from previous response |
| `sort` | string | varies | Sort field and direction (`field:asc` or `field:desc`) |

### Response

```json
{
  "data": [ ... ],
  "pagination": {
    "hasMore": true,
    "nextCursor": "eyJpZCI6MTIwfQ==",
    "previousCursor": "eyJpZCI6MTAwfQ==",
    "total": 1247
  }
}
```

**Cursor format:** Base64-encoded JSON containing the sort field value and ID of the last item. Opaque to the client — they should never decode or construct cursors manually.

#### Subscribable Events

**Member Events:**

| Event | Payload | Triggered When |
|-------|---------|----------------|
| `member.created` | Full member object | New account created (ThriveCart purchase or API) |
| `member.updated` | Member object with changed fields | Profile updated |
| `member.product_granted` | Member ID + product object | Product added to member |
| `member.product_revoked` | Member ID + product object + reason | Product removed (refund, expiration, admin) |
| `member.upgraded` | Member ID + old products + new product | Higher-tier product purchased |
| `member.onboarding_completed` | Member ID + completion timestamp | Onboarding wizard finished |
| `member.login` | Member ID + timestamp (debounced daily) | Member logs in |
| `member.deactivated` | Member ID + reason | Account deactivated |

#### Webhook Signature Verification

Partners verify webhook authenticity using HMAC-SHA256:

```javascript
const crypto = require('crypto');

function verifyWebhook(payload, signature, secret) {
  const timestamp = headers['x-bts-webhook-timestamp'];
  const signedPayload = `${timestamp}.${payload}`;
  const expected = crypto
    .createHmac('sha256', secret)
    .update(signedPayload)
    .digest('hex');
  return crypto.timingSafeEqual(
    Buffer.from(`sha256=${expected}`),
    Buffer.from(signature)
  );
}
```

#### Delivery & Retry

- Webhook deliveries are queued via BullMQ
- Timeout: 10 seconds per delivery attempt
- Retry schedule: 5 attempts with exponential backoff (30s, 2m, 15m, 1h, 6h)
- After all retries fail: subscription is marked as `failing`, admin is notified
- After 3 consecutive days of failures: subscription is auto-disabled
- Admin can view delivery history, see response codes, and retry individual deliveries
