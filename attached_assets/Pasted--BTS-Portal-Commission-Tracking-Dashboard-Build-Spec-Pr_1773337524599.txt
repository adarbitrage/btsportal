# BTS Portal — Commission Tracking Dashboard Build Spec

**Priority:** After Auth + ThriveCart + Onboarding + AI Chat + Community + Software & Tools
**Status:** Not started
**Depends on:** Entitlement system (done), Auth system, ThriveCart Webhook integration
**Reference:** BTS PRD v2.0, Sections 8.4, 9.2

---

## Context

The entitlement system defines four commission tiers:

```
commissions:entry    → 3-Month Mentorship
commissions:mid      → 6-Month Mentorship
commissions:premium  → 1-Year Mentorship
commissions:top      → Lifetime Mentorship
```

Each tier earns a different payout percentage when a member refers new customers who purchase through their unique referral link. The sidebar conditionally shows "Commissions" when any `commissions:*` entitlement is present. But there's no commission tracking UI or referral infrastructure built yet.

This spec builds the full affiliate commission system: referral link generation, cookie-based attribution, commission calculation on purchase events, payout tracking, and a member-facing dashboard showing earnings, referrals, and performance.

---

## How BTS Affiliate Commissions Work

```
REFERRAL FLOW:

1. Member (the "affiliate") shares their unique referral link
2. Prospect clicks the link → lands on a BTS landing page or checkout page
3. A tracking cookie is set attributing this prospect to the affiliate
4. Prospect purchases a BTS product via ThriveCart
5. ThriveCart (or the portal) attributes the sale to the affiliate
6. Commission is calculated based on the affiliate's tier and the product purchased
7. Commission is recorded in the portal and visible on the affiliate's dashboard
8. Payouts are processed on a schedule (monthly, manual, or via integration)

ATTRIBUTION:

Option A — ThriveCart Native Affiliates:
  ThriveCart has a built-in affiliate system. Each affiliate gets a ThriveCart 
  affiliate ID. ThriveCart handles cookie tracking, attribution, and commission 
  calculation. The portal reads commission data from ThriveCart via API or webhook.

Option B — Custom Tracking (Tapfiliate or Portal-Built):
  The portal generates referral links, handles cookie tracking, and passes 
  attribution data to ThriveCart as a custom field on checkout. Commission 
  calculation happens in the portal.

RECOMMENDATION: Use ThriveCart's native affiliate system for attribution and 
commission calculation. The portal provides the dashboard UI, referral link 
management, and performance analytics. This avoids building cookie tracking 
and attribution from scratch. If ThriveCart's affiliate features are insufficient, 
layer Tapfiliate on top later.
```

## Commission Tier Structure

Each mentorship tier earns a different commission percentage. The exact percentages are TBD but the system must support configurable rates per tier per product.

#### Commission Rate Schema

```sql
commission_rates
  id                  SERIAL PRIMARY KEY
  commission_tier     TEXT NOT NULL          -- 'entry', 'mid', 'premium', 'top'
  product_id          INTEGER REFERENCES products(id) NOT NULL
  rate_type           TEXT NOT NULL          -- 'percentage' or 'fixed'
  rate_value          DECIMAL(10,2) NOT NULL -- e.g., 30.00 for 30% or 50.00 for $50 flat
  is_active           BOOLEAN DEFAULT true
  created_at          TIMESTAMP DEFAULT NOW()
  updated_at          TIMESTAMP DEFAULT NOW()
  UNIQUE(commission_tier, product_id)
```

**Example rate structure (placeholder — Adam to finalize):**

| Product | Entry (3mo) | Mid (6mo) | Premium (1yr) | Top (Lifetime) |
|---------|-------------|-----------|---------------|----------------|
| Reserve Income ($47–$97) | 20% | 30% | 40% | 50% |
| Backroad ($47–$97) | 20% | 30% | 40% | 50% |
| Off-Market ($47–$97) | 20% | 30% | 40% | 50% |
| LaunchPad | 15% | 25% | 35% | 50% |
| 3-Month Mentorship | 10% | 20% | 30% | 40% |
| 6-Month Mentorship | 10% | 15% | 25% | 35% |
| 1-Year Mentorship | — | 10% | 20% | 30% |
| Lifetime Mentorship | — | — | 15% | 25% |

**Rules:**
- A member earns commissions based on their HIGHEST active commission tier
- If a member's mentorship expires, their commission tier drops to the next active level (or none if they have no active mentorship)
- Commission tier is resolved dynamically from entitlements, not stored as a static field
- Cells marked "—" mean that tier cannot earn commissions on that product (optional restriction, configurable)

## What to Build

### 1. Affiliate Profile & Referral Links

Every member with a `commissions:*` entitlement has an affiliate profile with unique referral links for each BTS product.

#### Affiliate Schema

```sql
affiliate_profiles
  id                  SERIAL PRIMARY KEY
  user_id             INTEGER REFERENCES users(id) NOT NULL UNIQUE
  affiliate_code      TEXT NOT NULL UNIQUE   -- short code (e.g., 'marcus-j', 'mj2026', auto-generated)
  custom_code         TEXT UNIQUE            -- member-chosen vanity code (optional, max 20 chars)
  thrivecart_aff_id   TEXT                   -- ThriveCart affiliate ID (if using TC native affiliates)
  paypal_email        TEXT                   -- payout destination
  payout_method       TEXT DEFAULT 'paypal'  -- 'paypal', 'bank_transfer', 'check'
  payout_threshold    DECIMAL(10,2) DEFAULT 50.00  -- minimum balance before payout
  tax_form_submitted  BOOLEAN DEFAULT false  -- W-9 or W-8BEN submitted
  is_active           BOOLEAN DEFAULT true
  created_at          TIMESTAMP DEFAULT NOW()
  updated_at          TIMESTAMP DEFAULT NOW()
```

**Affiliate code generation:**
- Auto-generated on first commission entitlement grant: lowercase first name + last initial + random 3 digits (e.g., `marcus-j-482`)
- Member can set a custom vanity code from their dashboard (e.g., `marcuswins`) — validated for uniqueness, alphanumeric + hyphens only, 3–20 characters
- The affiliate code is embedded in referral links

#### Referral Links

Each product has a referral link for the affiliate. Links are generated dynamically based on the affiliate code and the product's landing page URL.

**Link format:**

```
Option A — ThriveCart native:
https://checkout.thrivecart.com/tc_reserve_income/?affiliate=marcus-j-482

Option B — Portal-tracked:
https://buildtestscale.com/go/reserve-income?ref=marcus-j-482
  → Portal sets attribution cookie
  → Redirects to ThriveCart checkout with affiliate param

Option C — Landing page with tracking:
https://reserveincomesystem.com/?ref=marcus-j-482
  → Landing page JS reads ?ref param, stores in cookie
  → ThriveCart checkout link includes affiliate param from cookie
```

**For v1, implement Option B** — the portal acts as a redirect with cookie setting:

```
GET /go/:productSlug?ref=:affiliateCode

Logic:
1. Look up affiliate by code → if invalid, redirect without attribution
2. Set a first-party cookie: bts_ref={affiliateCode}, 30-day expiry
3. Redirect to the product's landing page URL (or direct to ThriveCart checkout)
4. When ThriveCart webhook fires for a purchase, check for bts_ref cookie 
   (passed as a custom field on the checkout URL)
```

#### Referral Link Table

```sql
referral_links
  id                  SERIAL PRIMARY KEY
  affiliate_id        INTEGER REFERENCES affiliate_profiles(id) NOT NULL
  product_id          INTEGER REFERENCES products(id) NOT NULL
  link_url            TEXT NOT NULL          -- the full referral URL
  short_code          TEXT UNIQUE            -- optional short URL code
  click_count         INTEGER DEFAULT 0      -- total clicks (denormalized)
  is_active           BOOLEAN DEFAULT true
  created_at          TIMESTAMP DEFAULT NOW()
  UNIQUE(affiliate_id, product_id)
```

Pre-generate referral links for every active product when the affiliate profile is created. New links are auto-created when new products are added.

### 2. Click Tracking

Track every click on a referral link for analytics.

#### Click Schema

```sql
referral_clicks
  id                  SERIAL PRIMARY KEY
  referral_link_id    INTEGER REFERENCES referral_links(id) NOT NULL
  affiliate_id        INTEGER REFERENCES affiliate_profiles(id) NOT NULL
  ip_address          TEXT
  user_agent          TEXT
  referer_url         TEXT                   -- where the click came from
  landing_url         TEXT                   -- where they were sent
  country             TEXT                   -- GeoIP lookup (optional, basic)
  device_type         TEXT                   -- 'desktop', 'mobile', 'tablet' (parsed from UA)
  created_at          TIMESTAMP DEFAULT NOW()
```

**Index:** `(affiliate_id, created_at)` for affiliate-specific click queries. `(referral_link_id, created_at)` for per-link analytics.

#### Click Tracking Endpoint

```
GET /go/:productSlug?ref=:affiliateCode

Logic:
1. Validate affiliate code → look up affiliate_profiles
2. Find referral_link for this affiliate + product
3. Log click to referral_clicks table (async — don't block the redirect)
4. Increment referral_links.click_count
5. Set attribution cookie: bts_ref={affiliateCode}, 30-day expiry, SameSite=Lax
6. Build redirect URL:
   - Product landing page URL + ?ref={affiliateCode}
   - OR ThriveCart checkout URL + ?affiliate={thrivecart_aff_id}
7. 302 redirect to the destination

Response time must be < 100ms — this is a redirect, not a page load.
```

**Deduplication:** Don't count multiple clicks from the same IP within 5 minutes as separate clicks. Check the most recent click from that IP for the same link and skip if within the window.

### 3. Commission Recording

When a purchase is attributed to an affiliate, a commission record is created.

#### Commission Schema

```sql
commissions
  id                  SERIAL PRIMARY KEY
  affiliate_id        INTEGER REFERENCES affiliate_profiles(id) NOT NULL
  referred_user_id    INTEGER REFERENCES users(id)  -- the customer who purchased
  product_id          INTEGER REFERENCES products(id) NOT NULL
  order_id            TEXT NOT NULL           -- ThriveCart order ID
  
  -- Sale details
  sale_amount         DECIMAL(10,2) NOT NULL  -- total sale price
  commission_tier     TEXT NOT NULL            -- tier at time of sale ('entry', 'mid', 'premium', 'top')
  commission_rate     DECIMAL(5,2) NOT NULL    -- rate applied (e.g., 30.00 for 30%)
  commission_amount   DECIMAL(10,2) NOT NULL   -- calculated payout amount
  
  -- Status tracking
  status              TEXT NOT NULL DEFAULT 'pending'
    -- 'pending'   → sale confirmed, waiting for refund window
    -- 'approved'  → past refund window, eligible for payout
    -- 'paid'      → payout processed
    -- 'reversed'  → sale was refunded, commission clawed back
    -- 'rejected'  → flagged as fraudulent or self-referral
  
  -- Payout tracking
  payout_id           INTEGER REFERENCES commission_payouts(id)
  approved_at         TIMESTAMP
  paid_at             TIMESTAMP
  reversed_at         TIMESTAMP
  
  -- Metadata
  notes               TEXT                    -- admin notes
  created_at          TIMESTAMP DEFAULT NOW()
  updated_at          TIMESTAMP DEFAULT NOW()
  
  UNIQUE(affiliate_id, order_id)  -- one commission per affiliate per order
```

#### Commission Calculation (in ThriveCart Webhook Handler)

Add to the existing ThriveCart webhook `order.success` handler:

```
After Step 5 (Grant Product Entitlements), add:

Step 5.5: CALCULATE AFFILIATE COMMISSION
  a. Check if the purchase has an affiliate attribution:
     - Look for bts_ref cookie value passed as custom field in ThriveCart payload
     - OR check ThriveCart's native affiliate field in the webhook payload
  
  b. If attributed:
     - Look up the affiliate profile by code
     - Validate: affiliate cannot earn commission on their own purchase (self-referral check)
     - Resolve the affiliate's current commission tier from their entitlements
     - Look up the commission rate for (tier + product) in commission_rates table
     - If no rate exists for this combination → skip (tier can't earn on this product)
     - Calculate: commission_amount = sale_amount × (rate_value / 100)
     - Create commission record with status 'pending'
  
  c. If not attributed:
     - No commission to record, skip
```

#### Refund Handling

Add to the existing ThriveCart webhook `order.refund` handler:

```
After revoking product entitlements, add:

Step: REVERSE COMMISSION
  a. Find commission record matching this order_id
  b. If found and status is 'pending' or 'approved':
     - Update status to 'reversed'
     - Set reversed_at timestamp
     - If commission was already paid: create a negative adjustment (see payouts)
  c. Log reversal to webhook_logs
```

### 4. Payout System

#### Payout Schema

```sql
commission_payouts
  id                  SERIAL PRIMARY KEY
  affiliate_id        INTEGER REFERENCES affiliate_profiles(id) NOT NULL
  amount              DECIMAL(10,2) NOT NULL  -- total payout amount
  commission_count    INTEGER NOT NULL        -- number of commissions in this payout
  method              TEXT NOT NULL            -- 'paypal', 'bank_transfer', 'check'
  destination         TEXT NOT NULL            -- PayPal email, bank details reference, etc.
  status              TEXT NOT NULL DEFAULT 'pending'
    -- 'pending'    → payout created, awaiting admin approval
    -- 'approved'   → admin approved, ready to process
    -- 'processing' → payment initiated
    -- 'completed'  → payment confirmed
    -- 'failed'     → payment failed (retry or manual intervention)
  
  reference_id        TEXT                    -- external payment reference (PayPal txn ID, etc.)
  admin_notes         TEXT
  approved_by         INTEGER REFERENCES users(id)
  approved_at         TIMESTAMP
  processed_at        TIMESTAMP
  completed_at        TIMESTAMP
  created_at          TIMESTAMP DEFAULT NOW()
  updated_at          TIMESTAMP DEFAULT NOW()
```

#### Payout Flow

```
1. APPROVAL WINDOW
   - New commissions start as 'pending'
   - After 30 days (configurable refund window), a nightly job moves 
     them to 'approved' if no refund occurred
   - This protects against paying commissions on refunded sales

2. PAYOUT GENERATION (monthly, admin-triggered or automated)
   - System aggregates all 'approved' commissions per affiliate
   - If total >= affiliate's payout_threshold: create a payout record
   - Link all included commissions to this payout (set payout_id)
   - Update commission status to 'paid'
   - Payout enters admin approval queue

3. PAYOUT PROCESSING
   - Admin reviews and approves the payout
   - Admin processes payment externally (PayPal mass pay, bank transfer, etc.)
   - Admin marks payout as 'completed' with reference ID
   - Affiliate receives notification (email via SendGrid)

4. For v1: Payouts are MANUAL
   - Admin generates payout reports
   - Admin processes payments outside the portal
   - Admin records the payment in the portal
   - Future: integrate PayPal Payouts API or Stripe Connect for automated payouts
```

#### Commission Approval Job

Nightly job (or admin-triggered endpoint):

```
POST /api/admin/commissions/run-approval

Logic:
1. Find all commissions where:
   - status = 'pending'
   - created_at < NOW() - 30 days (configurable refund window)
   - No matching refund in webhook_logs for this order_id
2. Update status to 'approved', set approved_at
3. Log count of approved commissions
```

#### Payout Generation

```
POST /api/admin/commissions/generate-payouts

Logic:
1. For each affiliate with approved, unpaid commissions:
   a. Sum all approved commission amounts
   b. If sum >= affiliate's payout_threshold:
      - Create payout record
      - Link commissions to payout
      - Update commission status to 'paid'
2. Return list of generated payouts for admin review
```

### 8. Fraud Prevention

#### Self-Referral Detection

```
On commission creation:
- If affiliate.user_id === referred_user_id → reject commission, status: 'rejected', 
  notes: 'Self-referral detected'
- Also check: if referred user's email domain matches affiliate's email domain AND 
  both are non-public domains (not gmail, yahoo, etc.) → flag for admin review
```

#### Cookie Stuffing / Click Fraud

```
- Track unique IPs per affiliate per day
- If an affiliate generates > 500 clicks in a single day with < 0.5% conversion → 
  flag for review
- If > 50% of clicks come from a single IP → flag for review
- Admin can pause an affiliate's commission earning pending investigation
```

#### Admin Fraud Dashboard

```
Flagged affiliates with:
- Suspiciously high click-to-sale ratio
- Self-referral attempts
- Same-domain referrals
- Abnormal click patterns
Actions: review, approve, reject individual commissions, pause affiliate
```

## What NOT to Build Yet

- Automated payouts via PayPal Payouts API or Stripe Connect (manual for v1)
- Two-tier affiliate commissions (earning on sub-affiliates' referrals)
- Custom landing pages per affiliate (dedicated affiliate microsites)
- Recurring commissions on subscription renewals (one-time commission per sale for v1)
- Real-time commission notifications via WebSocket (polling/page refresh is fine)
- Tapfiliate or third-party affiliate platform integration
- Affiliate application/approval process (auto-granted with mentorship entitlement)
- Commission dispute system (admin handles disputes manually for v1)
- GHL sync of commission data (log for now)
- Public-facing affiliate signup page (commissions are mentorship-exclusive, not a public affiliate program)

## Definition of Done

1. Affiliate profiles auto-created when a member gains any `commissions:*` entitlement
2. Unique referral links generated per affiliate per product with one-click copy
3. Click tracking logs every referral click with deduplication
4. `/go/:productSlug?ref=:code` redirect sets attribution cookie and logs click in < 100ms
5. ThriveCart webhook handler calculates and records commissions on attributed sales
6. Self-referral detection prevents members from earning commissions on their own purchases
7. Commission status lifecycle works: pending → approved (after 30-day window) → paid
8. Refunds correctly reverse associated commissions
9. Member dashboard shows earnings summary, per-link performance, commission history, and payout history
10. Leaderboard ranks affiliates by earnings with time period filters
11. Payout settings (PayPal, threshold, tax form) are editable by the member
12. Admin can manage rates, approve/reject commissions, generate and process payouts, and review fraud alerts
13. Commission rates are tier-aware: higher mentorship tiers earn higher percentages
14. Promotional resources (email swipes, banners, social posts) available to affiliates
15. No commission entitlement → "Commissions" nav hidden, all `/commissions/*` routes return 403
