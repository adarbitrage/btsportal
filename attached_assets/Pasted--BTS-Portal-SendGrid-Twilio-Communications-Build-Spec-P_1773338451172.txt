# BTS Portal — SendGrid + Twilio Communications Build Spec

**Priority:** After Coaching Booking System
**Status:** Not started
**Depends on:** Auth system, ThriveCart Webhooks, Onboarding flow
**Reference:** BTS PRD v2.0, Sections 15, 16

---

## Context

Multiple build specs reference email and SMS sending with "TODO: log to console for now" stubs. This spec replaces all those stubs with real SendGrid (email) and Twilio (SMS) integrations, plus builds the template system, sequence engine, and broadcast tools.

Every communication the portal sends flows through this system — transactional emails, marketing sequences, SMS reminders, and admin broadcasts.

---

## What to Build

### 1. Communication Service (Central Router)

A service module that all other parts of the portal call to send communications. No other code should import SendGrid or Twilio directly — everything goes through this service.

```typescript
// services/communications.ts

interface SendEmailParams {
  to: string;
  templateSlug: string;          // references an email_templates record
  variables: Record<string, any>; // template variable replacements
  userId?: number;               // for logging and unsubscribe tracking
  category?: string;             // 'transactional' or 'marketing'
}

interface SendSmsParams {
  to: string;                    // E.164 format: +15551234567
  templateSlug: string;          // references an sms_templates record
  variables: Record<string, any>;
  userId?: number;
}

// Queue-based sending (recommended — non-blocking)
async function queueEmail(params: SendEmailParams): Promise<void>
async function queueSms(params: SendSmsParams): Promise<void>

// Direct sending (for time-critical: password resets, verification)
async function sendEmailNow(params: SendEmailParams): Promise<void>
async function sendSmsNow(params: SendSmsParams): Promise<void>

// Batch sending (for broadcasts)
async function queueBroadcastEmail(params: BroadcastParams): Promise<void>
```

**All sends go through BullMQ** (except direct sends for time-critical auth flows). This prevents email/SMS sending from blocking API responses and allows retry on failure.

### 4. SendGrid Integration

#### Setup

```javascript
import sgMail from '@sendgrid/mail';
sgMail.setApiKey(process.env.SENDGRID_API_KEY);

async function sendEmail(params: SendEmailParams): Promise<void> {
  const template = await db.email_templates.findBySlug(params.templateSlug);
  if (!template || !template.is_active) throw new Error(`Template ${params.templateSlug} not found or inactive`);
  
  // Check unsubscribe for marketing emails
  if (template.category === 'marketing' && params.userId) {
    const unsub = await db.email_unsubscribes.findByUser(params.userId);
    if (unsub) {
      await logCommunication(params, 'skipped', 'unsubscribed');
      return;
    }
  }
  
  // Replace variables
  const subject = replaceVariables(template.subject, params.variables);
  const html = replaceVariables(template.body_html, params.variables);
  const text = template.body_text ? replaceVariables(template.body_text, params.variables) : undefined;
  
  // Add unsubscribe link for marketing emails
  if (template.category === 'marketing') {
    const unsubUrl = `${process.env.PORTAL_URL}/api/email/unsubscribe?token=${generateUnsubToken(params.userId)}`;
    params.variables.unsubscribe_url = unsubUrl;
  }
  
  const msg = {
    to: params.to,
    from: {
      email: template.category === 'transactional' 
        ? process.env.SENDGRID_FROM_TRANSACTIONAL    // noreply@buildtestscale.com
        : process.env.SENDGRID_FROM_MARKETING,       // hello@buildtestscale.com
      name: 'Build, Test, Scale'
    },
    subject,
    html,
    text,
    categories: [template.category, template.slug],
    customArgs: { userId: String(params.userId), templateSlug: template.slug }
  };
  
  await sgMail.send(msg);
  await logCommunication(params, 'sent');
}
```

#### SendGrid Configuration Requirements

- Dedicated sending domain with DKIM, SPF, and DMARC authentication
- Separate sender identities for transactional vs marketing
- Event webhook configured to receive: delivered, opened, clicked, bounced, unsubscribed, spam_report
- Suppression groups for CAN-SPAM compliance

#### SendGrid Event Webhook Handler

```
POST /api/webhooks/sendgrid

Receives delivery events from SendGrid. Update communication_log:

- 'delivered' → update status to 'delivered'
- 'open' → increment open_count, set first_opened_at
- 'click' → increment click_count
- 'bounce' → update status to 'bounced', flag email address
- 'unsubscribe' → create email_unsubscribes record
- 'spam_report' → create email_unsubscribes record + flag for review
```

### 5. Twilio Integration

#### Setup

```javascript
import twilio from 'twilio';
const client = twilio(process.env.TWILIO_ACCOUNT_SID, process.env.TWILIO_AUTH_TOKEN);

async function sendSms(params: SendSmsParams): Promise<void> {
  const template = await db.sms_templates.findBySlug(params.templateSlug);
  if (!template || !template.is_active) throw new Error(`SMS template ${params.templateSlug} not found`);
  
  // Check SMS opt-in
  if (params.userId) {
    const user = await db.users.findById(params.userId);
    if (!user.sms_opt_in) {
      await logCommunication(params, 'skipped', 'not_opted_in');
      return;
    }
  }
  
  const body = replaceVariables(template.body, params.variables);
  
  const message = await client.messages.create({
    to: params.to,
    from: process.env.TWILIO_PHONE_NUMBER,
    body
  });
  
  await logCommunication(params, 'sent', null, message.sid);
}
```

#### Twilio Configuration

- Dedicated phone number for BTS
- STOP keyword auto-handling enabled (Twilio manages opt-out automatically)
- Delivery status webhook: POST /api/webhooks/twilio
- One-way messaging only in v1 (no inbound message handling)

### 7. Unsubscribe System

CAN-SPAM compliance requires one-click unsubscribe in all marketing emails.

```sql
email_unsubscribes
  id              SERIAL PRIMARY KEY
  user_id         INTEGER REFERENCES users(id) NOT NULL UNIQUE
  unsubscribed_at TIMESTAMP DEFAULT NOW()
  source          TEXT NOT NULL              -- 'one_click', 'sendgrid', 'manual', 'spam_report'
  resubscribed_at TIMESTAMP                 -- if they opt back in
```

#### Unsubscribe Endpoint

```
GET /api/email/unsubscribe?token=xxx

Logic:
1. Decode the token to get userId
2. Create email_unsubscribes record (or update if exists)
3. Render a simple confirmation page: "You've been unsubscribed from BTS marketing emails. 
   You'll still receive important account notifications (receipts, password resets, etc.)."
4. Include a "Resubscribe" link
```

**Transactional emails are never affected by unsubscribe.** Members always receive: password resets, purchase receipts, session confirmations, ticket responses, and other critical account notifications.

### 8. Sequence Engine

Automated email/SMS sequences triggered by events (purchase, onboarding, inactivity, expiration). The engine manages timing, conditional logic, and step progression.

#### Sequence Schema

```sql
sequences
  id              SERIAL PRIMARY KEY
  slug            TEXT NOT NULL UNIQUE       -- 'onboarding_frontend', 'onboarding_mentorship', etc.
  name            TEXT NOT NULL
  description     TEXT
  trigger_event   TEXT NOT NULL              -- what starts this sequence
  is_active       BOOLEAN DEFAULT true
  created_at      TIMESTAMP DEFAULT NOW()
  updated_at      TIMESTAMP DEFAULT NOW()

sequence_steps
  id              SERIAL PRIMARY KEY
  sequence_id     INTEGER REFERENCES sequences(id) NOT NULL
  step_number     INTEGER NOT NULL
  channel         TEXT NOT NULL              -- 'email' or 'sms'
  template_slug   TEXT NOT NULL              -- references email_templates or sms_templates
  delay_minutes   INTEGER NOT NULL           -- delay from sequence start (not from previous step)
  
  -- Conditional logic
  condition_type  TEXT                       -- 'none', 'if_not_completed', 'if_not_logged_in', 'if_product_level'
  condition_value TEXT                       -- depends on condition_type
  
  -- Skip logic
  skip_if_completed TEXT                    -- skip this step if member completed this action
  
  sort_order      INTEGER NOT NULL
  is_active       BOOLEAN DEFAULT true
  created_at      TIMESTAMP DEFAULT NOW()

sequence_enrollments
  id              SERIAL PRIMARY KEY
  user_id         INTEGER REFERENCES users(id) NOT NULL
  sequence_id     INTEGER REFERENCES sequences(id) NOT NULL
  status          TEXT NOT NULL DEFAULT 'active'  -- 'active', 'completed', 'cancelled', 'paused'
  current_step    INTEGER DEFAULT 0
  started_at      TIMESTAMP DEFAULT NOW()
  completed_at    TIMESTAMP
  cancelled_at    TIMESTAMP
  cancel_reason   TEXT
  UNIQUE(user_id, sequence_id)
```

#### Sequence Processing

A BullMQ repeatable job runs every 5 minutes:

```
processSequences():

1. Find all active enrollments where status = 'active'
2. For each enrollment:
   a. Get the next step (step_number > current_step, is_active = true)
   b. Check if enough time has elapsed: NOW() - started_at >= delay_minutes
   c. Evaluate conditions:
      - 'if_not_completed': check if member has NOT completed onboarding (or other condition)
      - 'if_not_logged_in': check if member has NOT logged in since enrollment
      - 'if_product_level': check member's product level matches
      - If condition fails → skip this step, advance to next
   d. Check skip_if_completed:
      - If member has already completed the action (e.g., 'onboarding_complete') → skip
   e. If all checks pass: queue the email or SMS
   f. Update current_step to this step's step_number
   g. If no more steps remain: mark enrollment as 'completed'
```

### 9. Broadcast System

#### Broadcast Schema

```sql
broadcasts
  id              SERIAL PRIMARY KEY
  name            TEXT NOT NULL              -- admin label

  custom_subject  TEXT                       -- or custom subject (email only)

  delivered_count INTEGER DEFAULT 0

  created_by      INTEGER REFERENCES users(id)
  created_at      TIMESTAMP DEFAULT NOW()
  updated_at      TIMESTAMP DEFAULT NOW()
```

#### Segment Filters

```json
{
  "products": ["reserve_income", "backroad"],      // members who own these products
  "products_exclude": ["lifetime"],                 // exclude these product owners
  "has_entitlement": "community:access",            // members with this entitlement
  "onboarding_complete": true,                      // only onboarded members
  "last_login_within_days": 30,                     // active in last 30 days
  "registered_after": "2026-01-01",                 // registered after date
  "experience_level": ["beginner", "some_experience"], // from profile
  "sms_opt_in": true                                // for SMS broadcasts only
}
```

**Admin flow:**
1. Create broadcast (select channel, choose/create template, set segment filters)
2. Preview: system calculates estimated recipients and shows a sample of 5 matching members
3. Schedule or send immediately
4. Confirm (double confirmation for broadcasts to > 100 recipients)
5. System queues individual sends via BullMQ (batched to respect SendGrid/Twilio rate limits)
6. Results populate as deliveries and engagement events come in

### 10. Scheduled Sends (Reminders & Recurring)

Some communications are triggered by time-based events. A BullMQ repeatable job handles these.

```
processScheduledCommunications() — runs every 15 minutes:

1. COACHING CALL REMINDERS (group calls)
   - Find calls starting in 23–25 hours → send 24hr email reminder
   - Find calls starting in 55–65 minutes → send 1hr SMS reminder

2. 1-ON-1 SESSION REMINDERS
   - Find sessions starting in 23–25 hours → send 24hr email reminder
   - Find sessions starting in 55–65 minutes → send 1hr SMS reminder

3. MENTORSHIP EXPIRATION WARNINGS
   - Find user_products expiring in 29–31 days → send 30-day warning email
   - Find user_products expiring in 6–8 days → send 7-day warning email + SMS
   - Find user_products expired today → send expired email

4. SESSION FEEDBACK PROMPTS
   - Find completed sessions from 23–25 hours ago with no member_rating → send feedback prompt

5. COMMISSION NOTIFICATIONS
   - Find new commissions from last 15 min → send commission earned email
```

### 11. Bounced Email Handling

Protect sender reputation by handling bounces:

```sql
email_bounces
  id              SERIAL PRIMARY KEY
  user_id         INTEGER REFERENCES users(id)
  email           TEXT NOT NULL
  bounce_type     TEXT NOT NULL              -- 'hard' or 'soft'
  bounce_count    INTEGER DEFAULT 1
  first_bounced_at TIMESTAMP DEFAULT NOW()
  last_bounced_at  TIMESTAMP DEFAULT NOW()
  is_suppressed   BOOLEAN DEFAULT false      -- hard bounce = auto-suppress
```

**Rules:**
- Hard bounce → immediately suppress email address, no further sends
- Soft bounce → allow retries, suppress after 3 soft bounces in 7 days
- Suppressed addresses: all email sends check `email_bounces.is_suppressed` before sending
- Admin can view and manually unsuppress if address was corrected

| Location | Stub | Replace With |

| ThriveCart webhook (payment failed) | Log failure | `queueEmail('payment_failed', ...)` + `queueSms('payment_failed_sms', ...)` |

## Definition of Done

1. SendGrid sends real emails: transactional (receipts, password resets) and marketing (sequences, broadcasts)
2. Twilio sends real SMS: onboarding reminders, call reminders, payment alerts
3. All console.log stubs across the codebase are replaced with actual communication calls
4. Email templates are database-stored and admin-editable without code changes
5. SMS templates are database-stored and admin-editable
6. 3-day onboarding sequences run automatically for front-end and mentorship buyers
7. Nurture and re-engagement sequences run with conditional logic (skip if upgraded, skip if logged in)
8. Broadcasts can target member segments and be scheduled or sent immediately
9. Unsubscribe system works: one-click unsubscribe in marketing emails, transactional unaffected
10. Communication log records every send with delivery status and engagement tracking
11. Bounced emails are auto-suppressed to protect sender reputation
12. Scheduled reminders fire for coaching calls, 1-on-1 sessions, and mentorship expirations
13. Admin can manage templates, sequences, broadcasts, and view analytics
14. SMS respects opt-in: only sends to members who opted in during profile setup
